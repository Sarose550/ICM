# D12: Empirically-Optimal Block Size B as a Function of k (and n)

**Date:** 2025-07-22
**Hardware:** AMD Zen4 (EPYC, 185.8.107.239), governor=performance
**Status:** Complete

## Methodology

### Sweep Design

We measured the hybrid engine's wall-clock time for **every B ∈ {8, 16, 24, 32,
48, 64, 96, 128, 192, 256}** at each (n,k) cell across the grid:

- **n values**: {512, 1024, 2048, 4096, 8192, 16384, 32768}: standard
  benchmark grid (from `tools/sample_plans.c`'s `n_values[]`), skipping the
  smallest (256, too fast to resolve) and largest (65536, would take hours).
  32768 was only partially completed (k up to 275) due to time constraints,
  but 512–16384 are fully covered.

- **k values per n**: a hybrid grid of ~15–17 points per n:
  - **Dense bracket near the linear→hybrid crossover** (k≈275 on Zen4, per D10):
    {200, 240, 260, 275, 290, 310, 350, 400}
  - **Log-spaced grid from ~500 to k=n**: 6–8 additional points depending on
    n (fewer for large n to stay within time budget). Always includes k=n.

- **Q = 256** (standard for this project)
- **3 reps per (n,k,B) cell**, median recorded
- **Stacks**: uniform [1,100], seeded deterministically (srand(42))
- **Payout**: linear decay p[m] = n − m

The sweep bypasses `select_best_B()` entirely: for each (n,k) cell, it
directly calls `hybrid_ctx_create(n, S, k, B)` + `run_engine_ctx()` with
each B value, measuring wall-clock time via `clock_gettime(CLOCK_MONOTONIC)`.

For each (n,k) cell, we also record `icm_select_best_B(n,k)` (the cost
model's pick) for comparison; this is read once per cell, not per B.

**Time budget**: the full sweep took ~8 minutes for n=512 through n=16384.
n=32768 was truncated at k=275 because each cell at that n takes 600–2000ms
per timing, and completing all ~13 k values would have pushed the sweep
beyond ~45 minutes. The partial n=32768 data (k=200,240,260,275) is sufficient
to confirm the pattern holds at larger n.

### Raw Data

`results/b_optimal_sweep_zen4.csv`: 934 valid rows. Generated by
`tools/b_optimal_sweep.c`.

## Results: B-Selection Accuracy

### Headline

**B=32 is empirically optimal in 94 of 95 (n,k) cells (98.9%).**
The one exception is n=8192, k=350, where B=24 is 1.1% faster than B=32
(204.6ms vs 207.0ms), well within measurement noise for 3 reps.

The cost model (`select_best_B`) picks B=32 in all 95 cells. The cost model
is therefore **effectively perfect**: it matches the empirical optimum in
94/95 cells, and in the one mismatch it's only 1.1% off.

The U-curve is clean and symmetric near the minimum: B=16 is ~2.5% slower
than B=32 (too few blocks → more FFT work per block), B=64 is ~7.5% slower
(too many blocks → schoolbook block-build cost dominates). Beyond 64, the
penalty grows rapidly: B=256 is ~2.5× slower than B=32.

## Pattern Analysis: Does Optimal B Vary With k?

**No.** The empirically-optimal B is 32 for every (n,k) cell tested. It does
not vary with k (from k=200 to k=n), nor with n (from n=512 to n=16384).

The block size optimum appears to be a **hardware constant** on Zen4, driven
by the balance between:
- **Block-build cost** (O(n·B) schoolbook): grows with B, favoring smaller B
- **Tree FFT cost** (fewer blocks → fewer tree nodes): favors larger B
- **Leaf extraction** (O(n) with O(n/B) block overhead): weakly favors larger B

At B=32, these factors balance almost perfectly, and the balance point is
stable across the entire (n,k) parameter space.

### Why Doesn't It Vary With k?

The tree geometry (number of tree levels, polynomial sizes) does change with
k, but the *leaf-level* costs (block build + leaf extraction) dominate the
total in a way that's independent of k. Changing k primarily affects the
upper tree levels (near the root), where node counts are small relative to
the leaves, so the marginal impact on the B-optimal point is negligible.

## What About B Values Larger Than 64?

The D11 correctness probe showed that B up to 256 is numerically correct
(flat error, no degradation). But this performance sweep shows B > 64 is
**never faster** than B=32; in fact, B=256 is 2.5x slower on average.
There is no trend suggesting larger B would become optimal at larger n
(the penalty decreases slightly with n but remains >1.8× even at n=16384).

## Recommendation

**No change needed to `select_best_B()`'s candidate list or cost model.**

The current 6-value candidate list `{8, 16, 24, 32, 48, 64}` includes the
universal optimum (B=32) and the cost model correctly identifies it. The
candidate ceiling of 64 is fine: B=96+ is always slower, so there's no
reason to search beyond 64.

If anything, the candidate list could potentially be **shortened** (e.g.,
search only {16, 32, 48, 64} or even just {32}) since B=32 wins everywhere.
But the current 6-value search is cheap (it constructs tree contexts, not
full engine runs) and provides robustness against future hardware where the
optimum might shift. We recommend keeping it as-is.

### Caveat

This study used uniform stacks [1,100]. The block-build cost is independent
of stack values (it operates on transformed a_j = S_j / (S_j + y) values in
the quadrature loop), and leaf extraction is also stack-independent in cost
structure. There is no reason to expect the optimal B to depend on stack
distribution, and the project's other benchmarks confirm that hybrid engine
timing is insensitive to stack values.
